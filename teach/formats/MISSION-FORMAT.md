# MISSION.md Format

`MISSION.md` lives at `topics/<slug>/MISSION.md`. It is **optional and light** — write it only when the user has a concrete real-world reason to learn the topic that should steer sequencing. For pure curiosity, skip it; the MAP alone is enough.

When it exists, every "what to teach next" decision should be checkable against it.

## Template

```md
# Mission: {Topic}

## Why
{1-2 sentences. The concrete outcome the user is chasing — what changes in their work or life when they have this. Push past "to understand X" to the underlying goal.}

## Target depth
{passing familiarity | working knowledge | able to teach it} — and which areas matter most for that.

## Constraints
- {Time, deadline, prior background, learning preferences — anything that bounds the approach.}
```

## Rules

- **Optional.** A missing mission is fine. A vague one is worse than none — if the user can't say why, either interview briefly or skip it.
- **Keep it to a screen.** Past that, it's stopped being a compass and become a plan.
- **Target depth drives the bar.** "Passing familiarity" means `fluent` is enough for most areas; "able to teach it" means `can-teach` across the board. State it so the MAP knows when to stop.
- **Revise when the goal moves.** Missions shift as the user learns what they actually care about. Update in place; don't let a stale mission misroute sessions.
