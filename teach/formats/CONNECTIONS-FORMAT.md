# CONNECTIONS.md Format

`CONNECTIONS.md` lives at the **workspace root** — not inside any one topic — because its job is to link concepts *across* topics. It is built as topics accumulate. A connection is durable understanding: the moment a concept in one field clicks into a concept in another is when a pile of facts becomes a structure.

## Template

```md
# Connections

## {Concept A} ⇄ {Concept B}
_{topic-a} ⇄ {topic-b}_

{1-3 sentences: the actual link. Not "both relate to X" — the specific way understanding one deepens or reframes the other. Note if it's an analogy, a shared mechanism, a tension, or one being an instance of the other.}

Established: {YYYY-MM-DD} · Evidence: {the synthesis question the user answered to earn this}
```

## Rules

- **Cross-topic or cross-area only.** A link between two sub-areas of the same area is just the area's structure — that belongs in the lesson, not here. Reserve this file for non-obvious bridges.
- **Earn it with a synthesis question.** Don't assert connections *for* the user — ask the question that makes them draw the link, then record the one they actually made. The evidence line is the question.
- **Name the kind of link:** analogy (same shape, different domain), shared mechanism (literally the same thing underneath), tension (they pull against each other), or instance-of (one is a special case of the other). Naming the kind is what makes it reusable.
- **Prune false friends.** A connection that turns out to be a surface analogy with no real transfer should be removed — a wrong bridge mis-teaches both sides.
- **Revisit in later sessions.** Use established connections as interleaving fodder: re-test the link to keep both concepts alive.
