# GLOSSARY.md Format

`GLOSSARY.md` lives at `topics/<slug>/GLOSSARY.md`. It is the canonical language for the topic. Building it is itself part of learning: compressing a concept into a tight definition is evidence the user understands it — so a term earns its place only once the user can use it correctly, never the moment it's introduced.

## Structure

```md
# {Topic} Glossary

{One or two sentence description of what this glossary covers.}

## Terms

**Hypertrophy**:
Muscle growth driven by mechanical tension and metabolic stress over repeated training sessions.
_Avoid_: bulking, getting big

**Progressive overload**:
Systematically increasing the demand on a muscle over time — via load, volume, or intensity.
_Avoid_: pushing harder, levelling up
```

## Rules

- **Add a term only when the user can use it correctly** — not when it's first mentioned. The glossary records compressed knowledge; it is not a dictionary the user reads to learn. This is why it tends to lag the lessons.
- **Be opinionated.** When several words name the same concept, pick the best and list the rest under `_Avoid_`. That is how language compresses.
- **Keep definitions tight.** One or two sentences. Define what the term *is*, not how to do it.
- **Use the glossary's own terms inside definitions.** Once a term is in, prefer it everywhere — including inside other definitions. This is what makes later terms easier to grasp.
- **Group under subheadings** when natural clusters emerge (`## Mechanism`, `## Measurement`). A flat list is fine when terms cohere.
- **Flag field-level ambiguity.** If a term is used loosely in the wider field, note the resolution: "In this workspace, X always means Y."
- **Revise as understanding deepens.** A week-one definition may be wrong by week six. Update in place.
