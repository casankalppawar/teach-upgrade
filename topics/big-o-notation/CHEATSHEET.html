Big-O — quick reference

The distilled spine of the topic, as a card you revisit. (Synthetic example — shows what the skill produces.)
The one idea. Big-O measures how work grows with input size n, not seconds — because growth is a property of the algorithm, while seconds are a property of the machine. Keep only the dominant term; drop constants and lower-order terms. 3n² + 5n + 200 → O(n²).

The three notations

  • O — upper bound (grows at most this fast). The default; states a worst-case ceiling. A loose ceiling is still true — binary search is "O(n!)", just uselessly loose.
  • Ω — lower bound (grows at least this fast).
  • Θ — tight bound: O and Ω at once. The honest "this is the rate."

Reading it off code

  • Nesting multiplies · sequence adds. Two loops nested → n×n = O(n²). Two loops one-after-another → n+n = O(n).
  • Drop constants & lower-order terms. 3n²+5n+200 → O(n²).
  • Big-O is the worst case by convention; an early return changes the best case, not the O.
for a in items:        # n
    for b in items:    # n   → nested → O(n²)
        ...

for x in items: ...    # n
for x in items: ...    # n   → sequential → O(n)

Where it breaks

  • The ordering is for large n. For small n the lines cross — a tuned O(n²) can beat a sloppy O(n log n).
  • A class is a ceiling on growth, not a verdict on speed — constants you dropped can decide the real winner.
  • Cache, memory layout, and real hardware bend the theory. Asymptotics describe the limit, not every input.

Glossary

  • Order of growth — the dominant term after dropping constants; what actually decides scaling.
  • Asymptotic analysis — behaviour as n → ∞, where only the order of growth survives.
  • Logarithmic — rises by a fixed amount only when n doubles (the halving signature).
  • Linearithmic — n log n; the practical floor for comparison sorting.
  • Worst case — the input that makes it do the most work; what O usually describes.
